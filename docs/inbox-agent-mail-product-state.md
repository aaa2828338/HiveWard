# 收件箱与 Agent Mail 产品状态

## 目的

这份文档记录 HiveWard 收件箱/邮件行为的目标边界。它是产品状态说明，不是当前补丁的工程实现计划。

## 当前状态

- 收件箱目前主要由平台审批请求驱动。
- 生命周期代码调用审批服务时，平台可以直接创建收件箱条目。
- 部分收件箱正文来自 Agent 或 Manager 输出，但也有一部分由系统生命周期代码生成。
- 内部准备失败可能泄露成用户可见审批邮件，例如正文为 `Preflight Blocked` 的计划确认。
- 这会让收件箱看起来像系统在主动写信，而不是 Agent/Manager 给人的消息投影。

## 理想状态

- 收件箱是被动投影面。
- 只有 Agent 或 Manager 明确产出面向人的审批、请求、报告或消息时，才进入收件箱。
- 平台负责存储、线程、状态、权限、时间戳、路由和处理动作。
- Agent 或 Manager 负责内容、意图、措辞，以及为什么需要人回复。
- 内部编排检查、provider 失败、重试和生命周期 bookkeeping 应留在运行状态、日志或 Manager 上下文里；除非 Agent/Manager 明确把它写成面向人的请求。

## 边界

- 平台：路由、持久化、索引、展示和处理消息。
- Agent/Manager：决定什么时候需要人看到，并负责写消息本身。
- 系统生命周期：可以阻塞、重试或标记状态，但不应该自己创作 email-like 的用户消息。
- 审批能力是平台附加在 Agent/Manager 消息上的元数据，不是消息正文的替代品。

## 最终目标

用户对收件箱的感知应该是：“这些是 Agent 和 Manager 发给我的东西”，而不是“系统协调过程中自动生成的东西”。

自迭代场景里：

- 轮次计划确认应是 Manager 写出的计划请求。
- 发布报告确认应是 Manager 写出的报告请求。
- 复盘沉淀应发生在人确认或回复报告之后，使用已确认报告和人的补充输入。
- 内部预检判断属于 Manager 思考或运行上下文，不应该作为单独可见的邮箱作者。

## 当前非目标

这份文档不实现收件箱整体重构。当前工程改动只修局部展示和输出问题，并保留这份目标作为后续重构边界。
