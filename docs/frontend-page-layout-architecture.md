# Frontend Page Layout Architecture

## 背景

当前前端有一个明显问题：部分页面复用了运行页的 `.trace-page-grid` / `.trace-layout` 结构。运行页本身需要特殊分区，但这些类被当成通用页面底座后，会把不需要分区的页面也切成多段，造成收件箱这类页面出现“下半截黑屏”“内容被裁掉”“操作区不在同一个完整页面里”的视觉断裂。

产品上，HiveWard 的主页面不应该默认被拆成几块独立屏幕。默认页面应该是一个完整工作台：先有一个稳定的页面底座，再在底座上放不同业务组件。

## 产品原则

1. 每个一级导航页面都是一个完整页面，不是默认的上半区、下半区拼接。
2. 页面底座负责占满当前主工作区，业务组件只在这个底座里布局。
3. 页面内部可以有左右列、工具栏、列表、详情、对话、看板、抽屉和弹层，但这些都是组件层，不是重新切割整个页面。
4. 只有确实需要多区域同步观察的页面，才允许使用专属分屏结构。例如运行监控可以有运行步骤和输出区，但这个结构不能外溢成通用页面规则。
5. 所有长内容都应该在自己的内容容器里滚动，不应该让整页出现莫名空白，也不应该把底部操作区裁掉。

## 工程原则

### 1. 建立通用页面底座

建议新增或统一使用一套基础页面类，而不是继续复用 `.trace-*`：

```text
page-shell
page-header
page-body
page-surface
page-panel
page-scroll
```

推荐结构：

```tsx
<section className="page-shell">
  <header className="page-header">...</header>
  <main className="page-body">
    <section className="page-surface">...</section>
  </main>
</section>
```

基础语义：

- `page-shell`: 占满主工作区高度，定义标题区和内容区。
- `page-header`: 页面标题、筛选器、主操作。
- `page-body`: 页面主体，只负责承载业务布局。
- `page-surface`: 一个业务工作面，可以是全宽，也可以内部再分列。
- `page-panel`: 业务组件面板，例如列表、详情、编辑器。
- `page-scroll`: 明确的滚动容器。

### 2. 页面默认只使用两行

大多数页面应为：

```text
标题区
主体区
```

CSS 规则：

```css
.page-shell {
  height: calc(100vh - 32px);
  max-height: calc(100vh - 32px);
  display: grid;
  grid-template-rows: auto minmax(0, 1fr);
  overflow: hidden;
}

.page-body {
  min-height: 0;
  overflow: hidden;
}
```

不要把三段式、四段式结构写进通用页面类。多段布局必须是页面自己的业务类，例如 `.runs-monitor-layout`，不能叫 `.page-grid` 或 `.trace-layout` 这种看起来可复用的名字。

### 3. 业务页面使用自己的布局类

错误方向：

```tsx
<section className="page-grid trace-page-grid inbox-page-grid">
  ...
  <section className="trace-layout inbox-layout">
```

正确方向：

```tsx
<section className="page-shell inbox-page">
  <header className="page-header inbox-header">...</header>
  <main className="page-body inbox-body">
    <section className="inbox-layout">
      <InboxListPanel />
      <InboxConversationPanel />
    </section>
  </main>
</section>
```

`inbox-layout` 可以左右分栏，但它必须只控制收件箱主体，不应该继承运行页的高度和网格轨道。

### 4. 滚动必须局部化

列表、对话、JSON、Markdown、代码块、运行日志都可能很长。规则是：

- 页面本身不滚动，除非是移动端或内容型页面。
- 主体容器 `min-height: 0; overflow: hidden;`。
- 长列表和长正文使用自己的 `overflow: auto`。
- 底部操作栏不应该跟着内容滚走，除非产品明确要求。

收件箱目标结构：

```text
收件箱页面
├── 标题与统计
└── 主体
    ├── 收件列表：自己滚动
    └── 对话详情
        ├── 对话头
        ├── 消息线程：自己滚动
        └── 回复与审批操作：固定在详情底部
```

### 5. 禁止跨页面继承业务布局

以下类只能服务运行页，不应该给其他页面当底座：

```text
trace-page-grid
trace-layout
trace-column-shell
trace-issue-column
trace-output-column
run-report-layer
```

如果某页面确实和运行页视觉相似，也要复制为该页面自己的语义类，再抽取真正通用的底层规则。

## 当前已修问题

收件箱已从运行页三段式网格里收回：

- `inbox-page-grid` 覆盖为 `auto minmax(0, 1fr)` 两行。
- `inbox-layout` 占满主体高度。
- 左侧列表和右侧详情卡高度固定在主体内。
- 右侧对话线程自己滚动。
- 底部回复、批准、驳回按钮保留在详情卡底部。

这只是止血修复，不等于整体页面架构已经完成。

## 当前记录但未修问题

运行页在生命周期审批场景下还有明显前端状态错位：顶部已经显示“等待收件箱处理”，但下方“运行步骤 / 当前输出”区域可能仍显示“这个蓝图还没有运行记录 / 请选择一个节点”。这不是收件流程的主阻塞点，而是运行页把“审批前置阶段”和“节点运行阶段”混在同一个 trace 面板里展示，导致没有节点运行记录时页面看起来断裂。

后续前端重构时，运行页应把生命周期状态作为第一层状态展示：

- 如果运行正在等待审批，应优先展示审批摘要、收件箱入口、当前人类可读报告。
- 如果还没有节点运行记录，不应显示成空白错误态，而应显示“正在等待计划审批 / 尚未进入节点执行”的业务状态。
- `运行步骤` 和 `当前输出` 只负责节点级执行记录，不应该吞掉审批前置阶段的主状态。
- 这个问题归入运行页前端架构重构，不影响本轮先修收件箱审批链路。

### 运行页当前三段式展示问题

当前运行页临时拆成了多个展示区，基本能看出执行过程，但产品形态仍不对。后续应把运行页作为一个完整页面处理，而不是把“报告层 / 运行步骤 / 当前输出 / 产物列表”硬切成互相割裂的三段或多段。

后续重构方向：

- 页面底层仍应是一张完整运行工作台。
- 报告、步骤、当前输出、产物、审批状态都应是同一页面底座上的业务组件。
- 组件可以分栏、折叠、吸顶、滚动，但不能造成页面视觉断层或多套互相抢滚动的区域。
- 当前三段式展示只作为过渡状态接受，不作为最终布局标准。

### 产物打开链接问题

当前运行页和产物列表里的“打开”按钮存在路由问题：点击后会打开一个新的 HiveWard 网页，但没有正确跳转到真实产物、预览页、本地文件或可下载链接。

这属于附加功能，当前不阻塞主链路，可暂缓修复，但必须记录为后续前端任务：

- 区分 `storagePath`、`downloadUrl`、`relativePath`、预览 URL 和本地文件路径。
- Web 链接应打开正确的 HiveWard artifact 预览或下载地址。
- Windows 本地路径应展示为 `D:\...`，并提供复制或通过桌面壳打开的能力。
- “打开”按钮不能只打开一个空的 HiveWard 页面。
- 如果出于浏览器安全限制不能直接打开本地文件，应明确显示“复制路径 / 浏览器预览 / 下载”三个动作，而不是伪装成已打开。

## 后续重构清单

1. 建立通用 `page-shell` / `page-header` / `page-body` / `page-panel` / `page-scroll` CSS。
2. 将收件箱从 `.trace-*` 类完全迁出。
3. 将历史、配置、模型、Agent、Skills、频道等页面检查一遍，移除不必要的业务布局继承。
4. 运行页保留自己的专属布局，但类名应明确是运行页专用。
5. 重构运行页产物打开行为，确保“打开 / 复制路径 / 浏览器预览 / 下载”动作语义正确。
6. 每迁移一个页面，做桌面和窄屏截图验证。
7. 验证每个页面都没有底部黑屏、内容裁切、按钮被遮挡、双滚动条混乱。

## 验收标准

- 一级页面默认占满主工作区，不出现无意义空白断层。
- 页面主体只有一套明确滚动策略。
- 业务页面不再依赖其他业务页面的网格类。
- 列表、详情、编辑器、看板、对话等组件可以组合，但不会改变整页底座。
- 桌面宽屏、普通笔记本屏、移动窄屏都能看到完整页面结构和主要操作按钮。
