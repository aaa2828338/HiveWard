# HiveWard 产品原则

## 通用边界

1. HiveWard 首先是调度层。平台负责路由、生命周期、状态持久化、权限、安全边界和明确的产物发布契约；Agent 和 Manager 负责判断、策略、内容、取舍和表达。
2. 平台只做确定性的产品契约，不替模型做普通内容判断，不把模型输出包装成“看起来正确”的结果。
3. 限制模型或 harness 输出形状时，优先使用提示词、角色契约、输出 schema、示例和结构化上下文；不要用平台代码去猜测、改写或补救普通模型输出。
4. 不给 AI 擦屁股。坏的 AI 输出应该通过提示词修正、重新运行、拒绝或明确暴露，而不是由平台隐藏式兜底、归一化或兼容。
5. 如果问题来自生命周期、状态模型、调度边界或模块契约本身，就从底层设计修正。前端、后端、worker、store、shared 都要保持模块逻辑统一，避免用快捷补丁填补链路断点。
6. 保持 AI 的决策空间。是否继续调研、如何收敛需求、怎样解释风险、是否跳过某个动作等判断，应尽量交给 Agent 或 Manager；平台只提供清晰边界和必要上下文。
7. 产物归属必须清楚。只有本节点实际创建的新文件、页面、链接或内容，才能声明为本节点产物；检查、引用、验收、总结别人的产物时，只能写成引用对象，不能重新发布成自己的产物。
8. 变更完成后必须说明变更类型：提示词级、程序级或混合。提示词级改变模型行为；程序级改变调度、生命周期、持久化、权限、UI 或确定性产品契约；混合变更要分别说明两边的责任。
